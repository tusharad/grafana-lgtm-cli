### Task 1

Define the Client layer for prometheus. This code, shall receive the "query" as input and will return the result.

Inputs:
    1. query
    2. query_params (limit, start, end)
    3. client config:
        - base_url
        - timeout

Example, curl 'http://localhost:9090/api/v1/query?query=up&time=2015-07-01T20:10:51.781Z'

Output:
    1. Result (2xx) 
    2. 5xx service unavailable
    3. 4xx Bad request

The input would be text, but the output shall be serilizable json.

To formally verify this specific Prometheus Client layer, we will step down from the high-level functional composition used in the previous document and utilize **Type Theory and Total Functions**. 

Because this layer interacts with external network states (HTTP requests), the verification must prove that the function is **total**—meaning it safely handles all valid inputs and maps them to a defined, mutually exclusive output state (success or specific error codes), while satisfying your strict serialization constraint.

Here is the mathematical formalization and verification for the Prometheus Client layer.

### 1. Domain (Input Space) Definition

Let us define the input parameters as distinct mathematical sets:

* **Query String:** Let $Q$ be the set of all valid text-based PromQL queries.
* **Query Parameters:** Let $P$ be the set of all valid parameter tuples. 
    If $limit$ is a natural number ($\mathbb{N}$), and $start$ and $end$ belong to the set of valid ISO-8601 timestamps ($T$), then:
    $$P = \mathbb{N} \times T \times T$$
* **Client Configuration:** Let $C$ be the set of configuration tuples. 
    If $base\_url$ belongs to the set of well-formed URIs ($U$) and $timeout$ is a positive real number representing time in milliseconds ($\mathbb{R}^+$), then:
    $$C = U \times \mathbb{R}^+$$

The total input domain is the Cartesian product of these sets:
$$Domain = Q \times P \times C$$

### 2. Codomain (Output Space) Definition

The task specifies three distinct outcomes based on HTTP status codes. These are mutually exclusive, so we represent the output space as a disjoint union ($\uplus$).

* **JSON Response (2xx):** Let $J$ represent the set of all structurally valid, serializable JSON objects. Let $R_{2xx} \subset J$ be the specific subset of JSON objects matching the Prometheus API schema.
* **Client Error (4xx):** Let $E_{4xx}$ be the set of "Bad Request" states (e.g., malformed PromQL).
* **Server Error (5xx):** Let $E_{5xx}$ be the set of "Service Unavailable" or timeout states.

The total codomain is defined as:
$$Codomain = R_{2xx} \uplus E_{4xx} \uplus E_{5xx}$$

### 3. Function Definition

The Prometheus Client layer acts as a mapping function, $f_{prom}$, that executes the network request:

$$f_{prom}: (Q \times P \times C) \rightarrow (R_{2xx} \uplus E_{4xx} \uplus E_{5xx})$$



### 4. Formal Verification Proofs

To prove this design is mathematically correct and robust, we must satisfy two conditions based on your requirements: **Exhaustive State Handling (Totality)** and **Serialization Enforcement**.

#### Theorem 1: Totality of the Network Function
*Statement:* For every possible input tuple $i = (q, p, c) \in Domain$, the function $f_{prom}(i)$ resolves to exactly one defined state in the Codomain, preventing unhandled exceptions.

*Proof:*
1. The execution of $f_{prom}$ over an HTTP boundary acts as a piecewise function evaluated by the external Prometheus server and the local network configuration $c$.
2. Let $statusCode$ be the integer returned by the HTTP call. We can define the mapping as:
   * If $200 \le statusCode < 300$, the function maps to $R_{2xx}$.
   * If $400 \le statusCode < 500$, the function maps to $E_{4xx}$.
   * If $500 \le statusCode < 600$, or if the execution time exceeds $c_{timeout}$, the function maps to $E_{5xx}$.
3. Because the sets $R_{2xx}$, $E_{4xx}$, and $E_{5xx}$ are disjoint ($A \cap B = \emptyset$), the output state is mutually exclusive.
4. Because the HTTP specification guarantees a response within these blocks (or a local timeout yields $E_{5xx}$), there is no input $(q, p, c)$ that maps to an undefined state. The function is total. $\blacksquare$

#### Theorem 2: Strict Output Serialization
*Statement:* If the function evaluates to a successful state ($2xx$), the output is guaranteed to be serializable JSON.

*Proof:*
1. Let the result of $f_{prom}(q, p, c)$ be $y$.
2. Assume the network request is successful, meaning $y \in R_{2xx}$.
3. By our initial definition of the codomain, $R_{2xx}$ is a strict subset of $J$ ($R_{2xx} \subset J$), where $J$ is the universal set of valid serializable JSON objects.
4. Therefore, by the transitive property of subsets, if $y \in R_{2xx}$ and $R_{2xx} \subset J$, then $y \in J$. 
5. The output $y$ is mathematically proven to be serializable JSON when the request succeeds. $\blacksquare$

### Implementation Takeaway

This formalization dictates how your code *must* be structured. In strongly-typed languages (like Go, Rust, or TypeScript), this mathematical model perfectly translates to a **Result type** or an **Either Monad**. 

The code signature should look conceptually like this to honor the mathematics:
`f(query, params, config) -> Result<JSON, HTTPError>`

This ensures the compiler enforces the strict disjoint union ($R_{2xx} \uplus E_{4xx} \uplus E_{5xx}$) we just proved.
